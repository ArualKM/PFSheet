What's real vs. illustrative here

  • Real: the section list, groups, and labels are lifted verbatim from SECTION_GROUPS in components/character/editor/character-editor.tsx. Live Values bar mirrors the sticky top bar already shipped. The mobile bottom sheet mirrors the full-screen section navigator shipped 2026-07-01.
  • Illustrative: only 2–3 rows are wired with plain CSS transitions/JS to convey the expand/collapse feel (STR disclosure, the mobile sheet). Every other collapsed row is static — in the real build every section collapses/expands this way.
  • Ability score values, chip copy, and character name are placeholder flavor text, not a fixture.

Motion (motion/react) integration points

  • Chip → disclosure: shared layoutId per field; spring transition (type:"spring", stiffness:300, damping:30) — the tactile "pop" already described by --pf-ease-spring.
  • Active section card: layout animation on the surrounding stack so collapsing the previous active section and expanding the new one reflows the canvas smoothly instead of jumping.
  • Section switch (mobile arrows / rail click): AnimatePresence crossfade+slide, matching the existing pf-view-fade/pf-rise token durations.
  • Live Values bar: per-stat "flip" pulse (brief scale+color pop) when a recomputed value changes, so edits feel like they land instead of silently updating.
  • Mobile sheet: keep the shipped pf-sheet-in CSS keyframe as the base entrance; Motion only adds drag-to-dismiss (drag="y" + onDragEnd velocity check).
  • Gate everything on data-motion + prefers-reduced-motion via a small useReducedMotionPreference() hook wrapping Motion's own useReducedMotion() — collapses every spring above to an instant show/hide.

Contract this preserves

  • This is a presentation refactor only — useCharacterEditor's { draft, computed, status, error, canUndo, conflict, update, undo, resolveConflict } contract is untouched; no fork of the save loop.
  • All existing section editors (abilities-editor, combat-editor, spellcasting-editor, etc.) render inside the expanded card body — reuse them, don't rewrite them.
  • The one-line summary per collapsed section is a small derived-from-ed.computed helper per section (e.g. abilities → "STR 16 DEX 14 …"), not new server state.
